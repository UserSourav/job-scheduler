# API Documentation

Interactive, always-up-to-date docs are auto-generated by FastAPI and served
at **`/docs`** (Swagger UI) and **`/redoc`** whenever the backend is running
— e.g. http://localhost:8000/docs. What follows is a reference summary.

All endpoints are prefixed with `/api/v1`. Except where noted "no auth",
endpoints require a bearer token: `Authorization: Bearer <access_token>`.

## Auth

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/auth/register` | none | Create a user account |
| POST | `/auth/login` | none | Exchange email+password for a JWT access token |

## Organizations & Projects

| Method | Path | Description |
|---|---|---|
| POST | `/organizations` | Create an organization (creator becomes owner) |
| GET | `/organizations` | List organizations the current user belongs to |
| POST | `/projects` | Create a project under an organization |
| GET | `/projects` | List projects (optionally filter by `organization_id`) |
| GET | `/projects/{project_id}` | Get a single project |

## Retry Policies

| Method | Path | Description |
|---|---|---|
| POST | `/retry-policies` | Create a reusable retry policy (strategy, max_retries, delays) |
| GET | `/retry-policies` | List all retry policies |
| GET | `/retry-policies/{id}` | Get one retry policy |

## Queues

| Method | Path | Description |
|---|---|---|
| POST | `/queues` | Create a queue under a project |
| GET | `/queues` | List queues (optionally filter by `project_id`) |
| GET | `/queues/{queue_id}` | Get one queue |
| PATCH | `/queues/{queue_id}` | Update priority / concurrency_limit / retry policy |
| POST | `/queues/{queue_id}/pause` | Pause a queue (no new jobs will be claimed) |
| POST | `/queues/{queue_id}/resume` | Resume a paused queue |
| DELETE | `/queues/{queue_id}` | Delete a queue and all its jobs |
| GET | `/queues/{queue_id}/stats` | Job counts by status for this queue |

## Jobs (client-facing)

| Method | Path | Description |
|---|---|---|
| POST | `/jobs` | Create a job. `job_type` determines required fields (see below) |
| POST | `/jobs/batch` | Create several jobs sharing one generated `batch_id` |
| GET | `/jobs` | List/filter jobs — `queue_id`, `status`, `job_type`, `batch_id`, `page`, `page_size` |
| GET | `/jobs/{job_id}` | Get one job |
| GET | `/jobs/{job_id}/executions` | Full attempt history for a job |
| GET | `/jobs/{job_id}/logs` | Structured log entries for a job |
| POST | `/jobs/{job_id}/cancel` | Cancel a job not yet in a terminal state |
| POST | `/jobs/{job_id}/retry` | Manually re-queue a failed/dead-lettered/cancelled job |

### `job_type` and required fields

| job_type | Required fields | Behavior |
|---|---|---|
| `immediate` | — | Eligible for claiming right away |
| `delayed` | `delay_seconds` | Eligible after `now() + delay_seconds` |
| `scheduled` | `scheduled_at` | Eligible at an exact timestamp |
| `recurring` | `cron_expression`, `scheduled_at` (first fire) | The created job is a *template*; the scheduler process spawns runnable child jobs on each cron tick |
| `batch` | — (use `/jobs/batch`) | All jobs share a `batch_id` for grouped querying |

All job types accept an optional `idempotency_key` — resubmitting the same
key returns the original job instead of creating a duplicate — and an
optional `max_attempts` override for the queue's default retry policy.

## Jobs (worker-facing)

These are called by the worker process, not the dashboard or end users.

| Method | Path | Description |
|---|---|---|
| POST | `/jobs/claim` | Atomically claim up to `max_jobs` due jobs across the given `queue_names` |
| POST | `/jobs/{job_id}/start?worker_id=` | Mark a claimed job RUNNING, create a `JobExecution` |
| POST | `/jobs/{job_id}/complete` | Mark the execution succeeded, job COMPLETED |
| POST | `/jobs/{job_id}/fail` | Mark the execution failed; schedules a retry or moves to DLQ |

## Workers

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/workers/register` | none | A worker process registers itself on startup |
| POST | `/workers/{id}/heartbeat` | none | Periodic liveness + active-job-count report |
| POST | `/workers/{id}/drain` | none | Worker marks itself draining during graceful shutdown |
| POST | `/workers/{id}/offline` | none | Worker marks itself offline after draining completes |
| GET | `/workers` | required | List all registered workers (dashboard) |

> Worker-facing endpoints (`register`, `heartbeat`, `drain`, `offline`,
> `claim`, `start`, `complete`, `fail`) intentionally don't require a user
> JWT — workers authenticate via network-level trust (they only exist inside
> the deployment's private network) or, in a hardened production setup,
> a separate per-worker service credential. See `DESIGN_DECISIONS.md`.

## Dashboard aggregates

| Method | Path | Description |
|---|---|---|
| GET | `/dashboard/overview` | Job counts by status, worker counts by status, total queues |
| GET | `/dashboard/throughput?hours=24` | Completed vs. dead-lettered counts, bucketed by hour |
| GET | `/dashboard/dead-letter-queue` | All permanently-failed jobs, optionally filtered by `queue_id` |

## Errors

Errors follow FastAPI's standard shape:

```json
{ "detail": "Human-readable message" }
```

| Status | Meaning |
|---|---|
| 400 | Invalid state transition (e.g. cancelling a completed job) |
| 401 | Missing/invalid/expired token |
| 403 | Authenticated but not authorized (e.g. not an org member) |
| 404 | Resource not found |
| 422 | Request validation failed (missing required field for the job_type, etc.) |

## Pagination & filtering

List endpoints that return potentially large result sets (`GET /jobs`) use
`page` (1-indexed) and `page_size` (default 25, max 200) query params.
Filterable list endpoints accept the relevant foreign key / enum as a query
parameter (e.g. `GET /jobs?queue_id=...&status=failed`).
