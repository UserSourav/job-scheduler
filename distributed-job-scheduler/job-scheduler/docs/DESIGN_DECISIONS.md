# Design Decisions

This document explains the major engineering trade-offs made and what
would change for a larger-scale production deployment.

## 1. Atomic claiming: `SELECT ... FOR UPDATE SKIP LOCKED`, not a lock service

**Decision:** job claiming is one SQL statement — select candidate ids with
`FOR UPDATE SKIP LOCKED`, then `UPDATE ... WHERE id IN (...)` — instead of
introducing Redis/ZooKeeper/etcd for distributed locking.

**Why:** Postgres already gives us the exact guarantee we need (no two
transactions can lock the same row) with no extra moving parts to deploy,
monitor, or fail over. `SKIP LOCKED` means concurrent claimants never block
each other waiting on a row someone else is mid-claim on — they just move to
the next candidate. This was verified directly: `test_concurrent_claims_never_double_assign`
fires 8 concurrent claim requests against 10 jobs and asserts every job is
claimed exactly once.

**Trade-off:** this ties throughput to Postgres's write capacity. At very
high job-submission rates (tens of thousands of jobs/sec), a dedicated queue
broker (Kafka, SQS, RabbitMQ) would out-scale a single Postgres primary. For
the throughput this assignment targets, Postgres is simpler and gives us
transactional consistency between "claim this job" and "the job's own
metadata" for free — a broker would need a separate DB for that anyway.

## 2. Workers talk to the API, never the database directly

**Decision:** the worker process has zero SQLAlchemy/psycopg imports; every
interaction (claim, start, complete, fail, heartbeat) is an HTTP call to the
backend.

**Why:** this keeps workers deployable independently — different language,
different machine, scaled up/down without any DB connection-pool
coordination — and keeps all lifecycle/retry logic in one auditable place
(`app/services/job_service.py`) instead of duplicated across worker
implementations.

**Trade-off:** an extra network round-trip per lifecycle event compared to a
worker with a direct DB connection. Given job execution itself (the actual
work being scheduled) is virtually always slower than one HTTP round-trip,
this overhead is negligible in practice and buys significant deployment
flexibility.

## 3. Retry policies are a separate, reusable entity

**Decision:** `retry_policies` is its own table referenced by `queues`
(and overridable per-job via `max_attempts`), rather than columns inlined on
`queues`.

**Why:** organizations commonly want the same retry behavior across many
queues (e.g. "all payment-related queues retry 5x exponential"); extracting
the policy avoids duplicating and independently drifting that configuration.

**Trade-off:** one more join when reading a queue's effective retry
behavior. Worth it for the reuse and the ability to update a policy in one
place and have it apply everywhere it's referenced.

## 4. Recurring jobs: template + spawned children, not a single mutating row

**Decision:** a `RECURRING` job is a template row that never runs itself;
the scheduler process spawns a fresh child `Job` row (with its own id,
executions, and logs) each time the cron expression fires.

**Why:** this keeps each *execution* of a recurring job as a fully
independent, queryable record with its own retry/DLQ history — "did the
2am run last Tuesday succeed?" is a normal job lookup, not archaeology
through a mutated row's overwritten history.

**Trade-off:** the `jobs` table grows continuously for active recurring
jobs. At scale this is the natural candidate for the time-based partitioning
mentioned in `ER_DIAGRAM.md`.

## 5. Stale-job reclamation as a periodic sweep, not a distributed watchdog

**Decision:** a worker that dies mid-job (crash, OOM-kill, network
partition) without heartbeating leaves its jobs `CLAIMED`/`RUNNING`
indefinitely until the scheduler's periodic sweep (`reclaim_stale_jobs`)
puts them back in `QUEUED` after `WORKER_STALE_HEARTBEAT_SECONDS`.

**Why:** simple, centralized, and correct — no per-worker watchdog process
or lease-renewal protocol needed beyond "did this worker heartbeat
recently."

**Trade-off:** a crashed worker's in-flight job sits idle for up to the
stale-timeout window before reclamation (default 30s) rather than being
detected instantly. For jobs where at-least-once execution and bounded
staleness (not sub-second failover) is the requirement, this is an
acceptable and much simpler mechanism than distributed leases.

## 6. `create_all` at API startup, with Alembic available for real deployments

**Decision:** `app/main.py`'s lifespan calls `Base.metadata.create_all` on
boot so the project runs with zero setup steps. A full Alembic migration
environment (`alembic/`, autogenerated `initial schema` revision) is also
included and is what a real deployment should use going forward
(`alembic upgrade head`).

**Why:** for an assignment/demo context, "clone and run" with no manual
migration step is a better first-run experience. For a long-lived
production system, `create_all` cannot express schema *changes* (only
initial creation), so Alembic is the correct ongoing tool and is set up and
tested (`alembic revision --autogenerate`, `alembic upgrade head` both
verified against a live Postgres instance during development).

## 7. Worker-facing endpoints skip user JWT auth

**Decision:** `register`, `heartbeat`, `drain`, `offline`, `claim`,
`start`, `complete`, `fail` don't require the `Authorization: Bearer`
header that dashboard/user endpoints require.

**Why:** workers aren't end users and don't have (or want) a human account;
requiring a human-issued JWT would mean minting and rotating tokens for
infrastructure processes.

**Trade-off / what a hardened deployment would add:** a separate
machine-to-machine credential (e.g. a per-project API key, checked against
the `projects.api_key` column that already exists but isn't yet enforced
on these routes, or mTLS at the network layer) so an attacker on the
network can't submit fake completions. In this project's deployment model
(workers run inside the same private network/VPC as the API, with no public
ingress to worker-facing routes), network-level trust is the accepted
boundary — documented here explicitly rather than left implicit.

## 8. Dashboard is a static, dependency-free single-page app

**Decision:** the dashboard is one `index.html` with vanilla JS (no React
build step, no bundler) calling the REST API directly via `fetch`.

**Why:** zero build tooling means `docker-compose up` serves it immediately
with `python -m http.server`, and it's trivially deployable to any static
host (S3, Netlify, GitHub Pages) pointed at any backend URL via
`window.SCHEDULER_API_BASE`.

**Trade-off:** no component framework means more manual DOM string-building
for a larger app. For the scope of this dashboard (five views, mostly
read-only tables plus a handful of actions), this is a reasonable and very
low-overhead choice; a larger production dashboard would likely move to
React once the view count and interactivity grow past what template
strings comfortably support.

## What was deliberately left as a bonus / not implemented

Per the assignment's own evaluation weighting (engineering quality over
feature count), the following bonus features were not built, in favor of
making the core lifecycle, concurrency, and reliability mechanisms
correct and well-tested: workflow dependencies between jobs, rate limiting,
queue sharding, WebSocket live updates (the dashboard instead polls every
8s, which is simple and sufficient at this scale), full RBAC beyond
organization roles, and AI-generated failure summaries.
